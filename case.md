# UX-кейс: раздел Objects в RackTables

> **Что в комплекте**
> - `prototype/index.html` — улучшенный список Objects (фильтры, инкрементальный поиск, sticky-сортировка, density toggle)
> - `prototype/object.html` — улучшенная карточка объекта (single-page с коллапс-секциями, inline-edit, alert с CTA, кликабельная стойка)
> - `prototype/before/list.html`, `prototype/before/object.html` — текущие экраны с аннотированными проблемами
> - `assets/before-list.jpg`, `assets/before-object.jpg` — оригинальные скриншоты demo.racktables.org

---

## 1. Контекст и пользователь

**Кто:** инженеры дата-центра и сетевые администраторы (DCIM/NetOps). Работают по 4–8 часов в день, держат в фокусе десятки и сотни объектов одновременно. Это «power users» — им важна плотность данных, скорость и предсказуемость, а не «красивая» анимация.

**Где живёт основная работа:** раздел *Objects* — единый реестр всего железа (серверы, свитчи, роутеры, PDU, патч-панели, диски, шкафы) и связей между ними (порты, IP, NAT, размещение в стойках). Из Objects пользователь начинает почти любой сценарий.

**JTBD (Jobs To Be Done):** в Objects регулярно повторяются четыре задачи:

1. **«Найти конкретный объект»** — по имени, asset tag, IP, метке, стойке. Контекст: алерт/тикет → нужно открыть карточку.
2. **«Понять состояние объекта»** — где стоит, к чему подключён, что с ним не так, кто связан.
3. **«Внести точечную правку»** — поменять comment, обновить label порта, поправить cable ID, добавить tag. Изменение мелкое, но за день их десятки.
4. **«Добавить или подключить новый объект»** — новый сервер в стойку, новый порт к свитчу, новый IP — без потери контекста соседей.

## 3. Карта проблем

| # | Проблема | Где видно | Частота | Impact | Приоритет |
|---|---|---|---|---|---|
| P1 | Поиск спрятан в шапке, не фильтрует таблицу инкрементально; нет поиска по asset tag/IP в одном поле | Список Objects | Очень высокая (каждая сессия) | Высокий — TTC поиска +5–10 с | **High** |
| P2 | Нет фильтров по Type / Location / Status — есть только Tag filters | Список Objects | Высокая | Высокий — невозможно сегментировать | **High** |
| P3 | Колонки не сортируются, без визуальных индикаторов | Список Objects | Высокая | Средний | **Med** |
| P4 | Статус «Has problems» = только цвет строки, без иконки/легенды/причины | Список + карточка | Средняя | Высокий — слепая зона для дальтоников и в edge-кейсах | **High** |
| P5 | Карточка разнесена на 9 вкладок без иерархии важности | Карточка объекта | Очень высокая | Высокий — пользователь «угадывает», где живёт поле | **High** |

| P7 | «Has problems» баннер без причины и без CTA («что делать?») | Карточка объекта | Каждое появление | Высокий — теряется главный сигнал | **High** |
| P8 | Rackspace allocation справа не кликабелен — нельзя прыгнуть в карточку соседа | Карточка | Средняя | Средний | **Med** |
| P9 | Нет связных breadcrumbs к связанным объектам с превью | Карточка | Средняя | Средний | **Low** |
| P10 | Нет bulk-действий (в задании не приоритетно, но просится) | Список | Низкая, но болезненная | Средний | **Low / next** |

**Фокус решения** — H-проблемы P1, P2, P4 (список) и P5, P6, P7 (карточка). Эти шесть закрывают 80% болей по основным JTBD.

## 4. Гипотеза и принципы решения

**Гипотеза:** скорость и удобство Objects ограничены не нехваткой функций, а **архитектурой раскладки**: всё, что нужно делать «по сто раз в день», спрятано за переходами и формами. Если перенести 80% массовых операций в **inline-режим** и собрать **массовые фильтры на одной панели** — TTC основных JTBD сократится в 2–3 раза без удаления существующего функционала.

**Принципы:**

1. **Эволюция, не редизайн.** Сохраняем фирменные синие акценты, плотную таблицу и логику разделов. Меняем только то, что действительно мешает.
2. **Power-user first.** Плотность важнее воздуха. Density toggle (Comfortable / Dense) даёт обоим лагерям своё.
3. **Минимум переходов.** Каждое действие, которое выполняется > 1 раза в день — должно быть достижимо в 1–2 клика без потери контекста.
4. **Видимый статус.** Любая «проблемность» = иконка + цвет + причина + CTA. Никогда только цвет.
5. **Сохраняем функциональность.** Все вкладки и формы остаются — но как **fallback и для редких операций**, а не как основной способ.

## 5. Решение

### 5.1 Список Objects — `index.html`

Что меняем:

- **Единая панель фильтров сверху**: инкрементальный поиск (по name, label, tag, IP, location в одном поле) + чипы быстрых фильтров (Type, Location/Row, Status, Tag) с подсчётом совпадений на каждом чипе. Сохраняется существующий tag-фильтр, добавляются ранее отсутствующие срезы.
- **Сортировка по любой колонке** с визуальным индикатором (стрелка). Сохраняем все исходные колонки; добавляем «Status» как отдельную колонку с бейджем.
- **Статус — иконкой + бейджем + цветом одновременно** (а не только подсветкой строки). Левая 3-px полоса слева от строки + бейдж в новой колонке Status.
- **Density toggle** в футере таблицы: Comfortable (по умолчанию, 32 px) и Dense (28 px) — для тех, кто работает с 500+ строками.
- **Кликабельная Row/Rack** в строке — клик по локации применяет фильтр «только эта стойка/ряд».
- **Сохраняемые виды** (`⭑ Сохранить вид`) — пользователь сохраняет «production-серверы Row 1 с проблемами» и достаёт его одной кнопкой.
- **Сохранён всё тот же путь** в карточку объекта (клик по строке/имени) и сохранены `Add object`, breadcrumbs, верхняя нав-панель.

Скриншот «до» с разметкой проблем — `prototype/before/list.html`. Прототип «после» — `prototype/index.html`.

### 5.2 Карточка объекта — `object.html`

Что меняем:

- **9 вкладок → одна страница-обзор + коллапсируемые секции** (Summary, Ports, IP, NATv4, 802.1Q). Менее частые секции (NAT, VLAN, Files) — свёрнуты по умолчанию, но видны и доступны в один клик. Log/Files/Edit-properties вынесены в верхние быстрые кнопки.
- **Заголовок-«паспорт»** с ключевыми мета-полями в одну строку (Type, Label, Asset tag, HW, SW, Location) — пользователь видит идентификацию объекта без скролла.
- **Alert «Has problems» c причиной и CTA** («Создать задачу», «Снять флаг»). Раньше это была просто красная полоса.
- **Inline-edit для всех полей summary и таблиц** (порты, IPs, NAT-комментарии). Клик по значению → инлайн-инпут → Enter/Save → Toast «Сохранено». Все формы редактирования сохраняются как fallback (кнопка `✎ Edit properties` в шапке).
- **Inline-добавление порта/IP** через ghost-кнопку в footer субтаблицы (`+ Добавить порт`, `+ Добавить IP`). Без перехода на отдельную форму.
- **Кликабельная стойка справа**: клик по соседнему юниту = переход в карточку соседа. Активный объект подсвечен и обведён красным; легенда — снизу.
- **Connections panel** под стойкой: явно перечисляет linked-, peer- и routed-by-связи отдельным блоком, чтобы понять «к кому подключён» без чтения трёх таблиц.
- **Хлебные крошки до уровня стойки** (Main / Objects / London / Row 1 / Rack L01 / *London server 4*) — сохранены, как в оригинале, и расширены.

Скриншот «до» с разметкой — `prototype/before/object.html`. Прототип «после» — `prototype/object.html`.

## 7. Метрики успеха

Замеры «до» — оценочные, по реальным кликам в demo. «После» — фактические клики в прототипе.

| Сценарий | Кликов «до» | Кликов «после» | Δ |
|---|---:|---:|---:|
| 1. Найти `London server 4` среди 63 объектов | 1 + scroll (≈ 4–8 c) | Ввод 3–4 символов в поиск (≈ 2 с) | **−60–70 %** |
| 2. Отфильтровать «серверы с problems в Row 1» | Невозможно через UI | 3 клика по чипам | **из ∞ → 3** |
| 3. Изменить comment у объекта | 5: list → object → Properties → форма → Save | 3: list → object → клик по comment → Enter | **−40 %** |
| 4. Добавить порт к объекту | 4: object → Ports → форма → Save | 2: object → `+ Добавить порт` → Enter | **−50 %** |

**Рабочие метрики, которые я бы трекал в проде:**

- **Time-to-Find** в списке Objects (от открытия раздела до клика в строку) — цель: −50 %.
- **Time-to-Edit** (от открытия объекта до сохранения изменения) — цель: −40 %.
- **Доля inline-сохранений** против полных форм — цель: > 70 % правок проходят inline через 4 недели после релиза.
- **Error rate на сохранении inline** (валидация) — цель: < 3 %.
- **Использование сохранённых видов** — цель: ≥ 30 % активных пользователей за 4 недели.
- **Снижение повторных переходов «list ↔ card»** в одной сессии — цель: −30 %.

## 8. Риски, ограничения, next steps

**Риски и митигации:**

- *Привычка к табам.* Старшие операторы могут «не найти» Properties. Митигация: кнопка `✎ Edit properties` сохранена в шапке карточки, мигрирующая подсказка в первые 2 недели.
- *Inline-edit и валидация на бэке.* Текущий PHP-бэк RackTables валидирует через POST форм. Для inline — нужна точечная PATCH-обёртка над тем же кодом валидации. Не переписываем бэк, а оборачиваем существующие хэндлеры.
- *Плотность фильтров на маленьких экранах.* На < 1280 px фильтры схлопываются в `More…` с popover; основной поиск + 3 главных чипа всегда видны.
- *Сортируемые колонки на больших списках* (тысячи объектов) — нужна серверная сортировка/пагинация. Клиентская — только до ~500 строк.

**Что осознанно не делал в прототипе** (но указал бы как next steps):

1. **Bulk-actions** в списке (выбрать N объектов → массово сменить tag/местоположение/статус).
2. **Карта связей** объекта (graph view: linked, peers, routed-by) — сейчас даю Connections-панель как первый шаг.
3. **Сравнение объектов** (compare 2–3 карточки в сплите).
4. **Аудит-trail** правок прямо в карточке (кто/когда менял comment) — у RackTables есть Log, но он отдельной вкладкой.
5. **Перенос в Figma** — после согласования флоу в HTML.

## Как открыть прототип

1. Перейти в `~/Downloads/racktables-ux-case/prototype/`.

5. Сравнение «до»: `prototype/before/list.html` и `prototype/before/object.html`.
